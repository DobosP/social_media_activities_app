# Production-readiness & child-safety audit — 2026-05

A full verified audit of the codebase on `main`, reconciling the docs with the **actual
implemented state** and hardening the highest-risk paths before a public beta. It pairs
with [SAFETY](SAFETY.md), [THREAT_MODEL](THREAT_MODEL.md), [SECURITY](SECURITY.md),
[COMPLIANCE](COMPLIANCE.md) and [RELEASE_READINESS](RELEASE_READINESS.md).

## Method

Every Django app was mapped against the roadmap, then the code was audited across 11
dimensions (child-safety invariants, object-level authz, injection/SSRF/upload, the
E2EE + EUDI cryptography, privacy/GDPR, correctness/concurrency, production config,
DB/performance, rate-limiting, API contract, and test coverage). **Every critical/high
finding was independently re-verified against the code** to filter false positives.

- **62 dimension findings** + ~20 app-level findings.
- Of the 23 critical/high findings, **0 were refuted** on adversarial re-check (3
  confirmed-critical, 12 confirmed-high, 6 downgraded after verification).
- The full, sortable register is at the end of this document.

## The headline: docs vs. reality

The roadmap and release docs were **internally inconsistent and over-optimistic**. The
single most important discrepancy:

> A complete **D10 secure-messaging subsystem** (`apps/messaging`, ~2.8k LOC of E2EE
> direct & group chat) is built and URL-wired, yet `ROADMAP.md`/`WORKBOARD.md`/
> `PHASE_2_PLAN.md` never mention it and `RELEASE_READINESS.md` still asserted *"no DM
> system exists ✅ by design."* `SAFETY.md`/`MESSAGING.md` were updated for it — so the
> docs even contradicted each other.

Other drift: `apps/web` (server-rendered UI), `apps/notifications`, and
`apps/recommendations` shipped but were absent from the roadmap; the feature-traceability
table still marked shipped D2 items "⏳ pending"; D3's *"create an activity via the API"*
is a **dead route** (`ActivityViewSet` is read-only → `POST` 405) with a false-green test;
the user-place **quorum has no API** (admin-only); D8 booking is a **demo stub** (its
definition-of-done is unmet); and P6 i18n is marked "built" though the web templates carry
**zero `{% trans %}` tags** (the UI is English-only).

## Wave 0 — fixed in this change (child-safety & security launch-blockers)

All on branch `claude/wave0-safety-hardening`, with negative regression tests. *(This
environment has no Django/DB installed, so the suite could not be run locally — every
touched file passes `py_compile`/`compileall`, the tests mirror existing passing patterns,
and CI runs the real suite on push.)*

| ID | Fix | Files |
|----|-----|-------|
| **CFG-1** (crit) | `prod.py` now **fails closed** on `SECRET_KEY` — required from env, rejects the dev sentinel (no more world-readable signer). | `config/settings/prod.py` |
| **UPLOAD-1** (crit) | Media scanning **fails closed**: scans the **original** bytes (not the Pillow re-encode), and refuses uploads unless an *effective* scanner is configured (`MEDIA_REQUIRE_SCANNER`, `ImageScanner.is_effective()`). | `apps/media/{services,scanning}.py`, `config/settings/{base,dev,test}.py` |
| **SAFE-1** (crit) | Messaging now enforces the **parental-consent gate**: `can_message()` and `register_public_key()` require `can_participate()` for both parties — a non-consented under-16 can no longer register keys or run E2EE chats. | `apps/messaging/services.py` |
| **ACCT** (crit, map) | Added a **parental-consent grant/revoke** service + API (`POST/DELETE /api/accounts/wards/<id>/consent/`); `link_guardian()` now requires the guardian be a verified **adult**. Minor onboarding no longer needs Django admin. | `apps/accounts/{services,views,urls}.py` |
| **SAFE-4** (high) | Consent is **re-checked at access time**, not just at join: chat thread read/write and activity-thread posting now call `can_participate()`. | `apps/chat/services.py`, `apps/social/services.py` |
| **guardian-post** (high, map) | An accompanying guardian (adult) can **no longer post** into a children's activity thread (`post_to_thread` excludes the `GUARDIAN` role) — matching the read-only observer rule. | `apps/social/services.py` |
| **chat-block** (high, map) | Blocking is honoured inside a shared thread: a user blocked-with the activity owner is denied chat access. | `apps/chat/services.py` |
| **CONC-2** (high) | Cohort is **re-validated** in `accept_invite()` and on `post_message()` send, so a cohort change between invite/accept (or after joining) can't create a cross-cohort adult↔minor channel. | `apps/messaging/services.py` |
| **CONC-3 / PRIV-4** (high) | A **guardian observer is removed** when its last ward leaves/declines/is removed, and `revoke_guardian()` now cascades into messaging — an adult can't keep reading a child's E2EE conversation after the consent basis ends. | `apps/messaging/services.py`, `apps/accounts/services.py` |
| **SAFE-6** (med) | Added negative regression tests asserting these invariants **cannot** be violated. | `apps/{messaging,chat,social,accounts}/tests/` |

## Recommended next waves (not yet done)

- **Wave 1 — production hardening:** `CACHES`/Redis so rate-limits + channel layer are
  global, not per-process (`CFG-2/3`, `RATE-1`); brute-force protection on login/register
  (`RATE-2`); a **scheduler** for `purge_messaging`/`purge_chat`/`lift_suspensions`
  (`PRIV-3`); a GDPR **erasure** path + media-blob cleanup (`PRIV-1/2`); stored-URL XSS
  scheme validation (`XSS-1`); join-by-vote **capacity** re-check (`CONC-1`); per-scope
  throttles (`RATE-3`); donation-webhook auth (`API-1`).
- **Wave 2 — correctness/perf/API + tests:** the medium/low register items (pagination,
  indexes incl. pgvector ANN, EUDI holder-binding/replay, SSRF host-allowlisting, API
  validation), and a CI test that prod resolves a real identity provider (`TEST-1`).
- **Wave 3 — product & docs:** expose the activity-create + place-quorum APIs; make
  moderation **remove/warn** actually hide content; finish RO/EN translation; wire error
  tracking (Sentry). And keep this audit's reconciled docs current.

## Residual notes on the Wave 0 fixes

- **Establishing** a guardianship link still flows through the verified parental-consent
  identity process (pending the live EUDI wallet, ~Dec 2026); Wave 0 only made the
  **consent grant/revoke** self-service for an already-linked guardian.
- chat blocking is enforced against the **activity owner** (consistent with discovery);
  member-vs-member blocking inside a shared thread remains a product decision.
- The fail-closed upload gate means **photo uploads are disabled** until
  `MEDIA_CSAM_HASH_BLOCKLIST` is populated or `MEDIA_IMAGE_SCANNER` points at a real
  service (this is the intended safe default for a minors' platform).
