# Production-Hardening & Legal-Deployment Plan — Social Activities Platform (Romania/EU, minors)

*Lead-architect synthesis, verified against `main @ 257cb44` on 2026-05-29. Every claim below was re-checked against current code, not docs. Citations are `file:line`. "Wave 0" = already committed and confirmed present.*

---

## 1. Executive summary

- **The engineering child-safety core is genuinely good and largely real**, not aspirational: cohort isolation, the `can_participate` consent gate wired into booking/social/chat/messaging, fail-closed media scanning, EXIF/GPS strip, hash-chained audit log, E2EE messaging with a single anti-grooming chokepoint (`can_message`), and a cryptographically sound EUDI ES256 verifier (`algorithms=["ES256"]`, audience+nonce+exp binding). These are tested and confirmed.
- **But the minor user journey is dead-ended in production.** `link_guardian` (the only function that creates a `GuardianRelationship`) has **zero non-test callers** — verified: every caller is under `apps/*/tests/*` (`apps/accounts/services.py:51`). No URL, no API, no web view creates the link. An under-16 can register but can *never* reach `can_participate=True`. This nullifies the Wave 0 consent flow.
- **Age assurance never expires in practice.** `can_participate` (`apps/accounts/services.py:40-47`) checks only the `is_identity_verified` boolean — it never consults `AgeAssurance.expires_at`. A child who ages out of a band, or a 13-month-old proof, keeps cohort + verified status forever.
- **The deploy is a single sleeping process with no Redis, no shared cache, no worker/cron, no error tracking, no DB backups, and no EU region pin** (`render.yaml` — confirmed: one `type: web` service, no `region:`, no cron, `channels-redis`/`redis`/`sentry` absent from requirements). Real-time fan-out, durable rate limiting, retention purges, and suspension auto-lift therefore do not work in any multi-process or even long-lived deploy.
- **Several launch-blocking GDPR/DSA duties have no code at all:** no Art. 17 erasure path (`apps/accounts/urls.py` exposes only me/wards/consent/verify-age), media blobs orphan on user delete (`uploader on_delete=CASCADE`, `apps/media/models.py:20`), no privacy/terms pages (none in `apps/web/urls.py`), no DSA statement-of-reasons (`take_action` never notifies; `apps/safety/services.py:116-146`), and retention purges never run.
- **Moderation REMOVE/WARN is a no-op.** `take_action` only mutates state inside `if action in (SUSPEND, BAN)` — reported content stays fully visible after a moderator "removes" it. There is no `is_hidden` field on Post/Activity.
- **Security gaps that are exploitable today:** decompression-bomb OOM on upload (no `MAX_IMAGE_PIXELS`, full pixel decode before cap — `apps/media/processing.py:32-37`); donation webhook unauthenticated by default + no Stripe signature (`apps/donations/views.py:64-66`); stored XSS via ingested `place.website`/`event.url` rendered raw into `href`; no login/register brute-force protection.

### Go/No-Go verdict

**NO-GO for a public beta with real minors today.** Two independent classes of blocker exist:

1. **Functional/safety:** minors cannot be onboarded at all (no guardian-link path), and verification never expires.
2. **Legal:** no erasure, no privacy policy/terms, no statement-of-reasons, no DPIA/DPO/ROPA, no EU residency pin, retention never runs, no real CSAM matcher (uploads are silently disabled in prod by fail-closed gate).

A closed pilot with **adults only** (or synthetic minor accounts in staging) is feasible after Wave 1's technical subset; onboarding real minors requires Wave 1 *plus* the human legal artifacts (DPIA, DPO, privacy policy, RO counsel sign-off).

### Single biggest risks (ranked)

| # | Risk | Why it dominates |
|---|------|------------------|
| 1 | No production guardian-link path (CS-1) | 100% of the minor journey is broken; the product's reason-for-being doesn't function for its target users. |
| 2 | Self-declared age trusted if `EUDI_SANDBOX=True` (CS-3) | One env flag from total age-assurance bypass; sandbox issuer signs any claimed age. |
| 3 | No erasure + media-blob orphaning (GDPR Art. 17) | Hard legal blocker; children's image bytes survive deletion. |
| 4 | Single-process deploy with no Redis/cache/worker | Real-time, rate-limiting, and retention are all silently broken or non-durable. |
| 5 | Decompression bomb + unauthenticated donation webhook | Remotely triggerable DoS and donation-record tampering, today. |

## 2. Architecture overview (as-is)

**Stack:** Django 5.2 LTS + DRF + GeoDjango/PostGIS (single datastore — PostGIS *and* pgvector live in one Postgres), Channels/daphne ASGI for WebSockets, WhiteNoise for static, server-rendered `apps.web` UI co-existing with the REST API over the same domain services.

**Subsystems (all confirmed wired end-to-end, not dead routes):**

| Cluster | Apps | Role |
|---|---|---|
| Identity | `accounts` | Custom User (age_band + cohort, **never a birthdate** — `models.py:67`), EUDI OpenID4VP verifier (real ES256), consent gate `can_participate`, pluggable `IDENTITY_PROVIDER` (dev stub default). |
| Social-RT | `social`, `chat`, `messaging` | Cohort-scoped activities/join-by-vote; per-activity live chat (server plaintext); zero-knowledge E2EE direct/group messaging (D10) — server stores only ciphertext + per-recipient wrapped keys. |
| Safety-media | `safety`, `media`, `notifications` | Reports/moderation/blocking/hash-chained audit; one-profile-photo + thread-photo pipeline (validate → EXIF-strip → fail-closed CSAM hash scan → signed serving); in-app-only notifications. |
| Data-geo | `places`, `ingestion`, `events`, `discovery`, `recommendations` | PostGIS venues + OSM/Overture/iCal ingestion (manual commands); pgvector content-based recommendations (declared interests + joins only, **no behavioural tracking**). |
| Platform | `config`, `ops`, `donations`, `booking`, `web` | Layered settings (base/dev/test/prod), `/healthz`, Stripe/deep-link donations, deep-link booking, server-rendered UI, Render/Docker/CI. |

**Data store:** one Postgres (PostGIS + pgvector extensions in migrations). No Redis, no separate cache, no object-store CDN.

**Real-time:** `config/asgi.py` → single `ProtocolTypeRouter` → `AllowedHostsOriginValidator(AuthMiddlewareStack(URLRouter(chat + messaging)))`. Channel layer defaults to **`InMemoryChannelLayer`** (`config/settings/base.py:252-259`) — process-local groups. Consumers authorize **only at connect** and cache the conversation/thread; `conversation_message` fans out to every group member with **no per-recipient recheck** (`apps/messaging/consumers.py:18-47`).

**Deploy:** Render Blueprint — one free `type: web` daphne process + one free PostGIS DB. Sleeps after ~15 min idle. `preDeployCommand` = migrate only. **No region pin, no cron/worker, `DJANGO_SECRET_KEY` generated** (`render.yaml`). Prod media defaults to `FileSystemStorage` (`config/settings/prod.py:59-62`) — ephemeral on Render.

**Anti-abuse:** `safety.services.allow_action` + DRF throttles, both backed by Django's default cache → **LocMemCache** (no `CACHES` defined anywhere — confirmed). Per-process, non-atomic, reset on every cold start.

### What breaks at scale

- **~100 concurrent users (single free instance):** App "works." N+1 COUNT feed latency and media bytes through the worker are noticeable; cold-start kills idle sockets; rate-limit counters reset on each wake. Tolerable for a closed pilot, painful UX.
- **~1k users:** Single 512MB/1-vCPU worker + free Postgres saturate. The N+1 feed and unindexed `Block.blocked` dominate DB load; the audit-chain lock caps messaging throughput; unbounded tables (no purge) start bloating queries. You *must* fix PERF-1/3, add Redis cache, and add the scheduler. Real-time still works only because you can't yet scale out.
- **~10k users:** Architecturally impossible on the current shape. The moment you add a second daphne worker, real-time fan-out splits (SCALE-1) and rate limits go N× looser (SCALE-3). Requires Redis channel layer + shared cache, PgBouncer, media off the worker (CDN/presigned), ANN index, and the audit-chain redesign. The "horizontally scalable" claim in `docs/ARCHITECTURE.md:91` is **false** for the real-time and rate-limit layers (SCALE-8).

## 4. Target production architecture

**Current:** single daphne process + free PostGIS + LocMemCache + InMemoryChannelLayer + FileSystemStorage. No worker, no CDN, no observability, no backups.

**Target (cheap-but-correct, nonprofit-friendly):**

| Layer | Current | Target | Notes / cost posture |
|---|---|---|---|
| Compute | 1 free daphne (sleeps) | 1 paid Render web (Frankfurt, no sleep) running daphne `--proxy-headers` | Smallest paid tier removes sleep, fixes cold-start socket death + counter resets. |
| Channel layer | InMemory | **Redis** (`channels-redis`) via `REDIS_URL` | Shared with cache. Render Key Value / Upstash free-ish tier. Unblocks scale-out + cross-process WS eviction. |
| Cache / throttles | LocMemCache | **Redis** (django-redis), atomic limiter | Same Redis. Durable rate limits across restarts/instances. |
| Scheduler | none | **Render Cron** (or GitHub Actions scheduled workflow hitting a management trigger) running purge_messaging/purge_chat/lift_suspensions/send_activity_reminders | Required for GDPR retention + suspension auto-lift. Cron is paid on Render; the free fallback is an external scheduler. |
| Media | FileSystemStorage (ephemeral) | **S3/R2 in an EU region** + presigned GET URLs (+ optional CDN) | Cloudflare R2 has a generous free tier and EU jurisdiction options. Removes bytes from the worker; satisfies residency. |
| DB | free PostGIS (no backups, expires) | **Paid Render Postgres (Frankfurt) with automated backups/PITR** + PgBouncer if scaling out | Free tier has no backups and a limited lifespan — unacceptable for consent/audit/key data. |
| Observability | `/healthz` + admin stats | **Sentry (free tier) + structured LOGGING** | Currently 0% of unhandled prod exceptions are captured. |
| Region | unset (Render default) | `region: frankfurt` for web + DB; `MEDIA_S3_REGION` EU | EU residency for minors' data (GDPR Ch. V). |

**Minimum to leave free tier safely (recommended first paid step):** one paid web instance + Redis (cache **and** channel layer) + EU object storage + a paid DB with backups + a cron. This is the cheapest topology that is *correct* (real-time, durable limits, retention, residency, backups). Estimated low-double-digit EUR/month — appropriate for a nonprofit beta.

**Boot assertions to add (prevent regression):** prod must reject `InMemoryChannelLayer`, reject LocMemCache, reject the dev `IDENTITY_PROVIDER`, and warn loudly if `EUDI_SANDBOX=True` with `DEBUG=False` — mirroring the existing `SECRET_KEY` fail-closed pattern (`prod.py:16-21`).
